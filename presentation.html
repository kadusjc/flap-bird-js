🎯 Project Overview

A Flappy Bird-style game running entirely in the browser, enhanced with a YOLOv5 neural network that performs real-time object detection on the game canvas.

Goal: Demonstrate how machine learning models can run client-side in a browser using TensorFlow.js, with zero backend infrastructure.
  • Pure HTML5 Canvas + vanilla JavaScript (ES Modules)
  • AI runs in a Web Worker (separate thread)
  • Model: YOLOv5n (nano) — the smallest & fastest YOLOv5 variant
  • No server, no API calls — everything runs on the user's device

🔄 The Game Loop

The heartbeat of every real-time game. Runs ~60 times per second.

// main.js
function loop() {
  game.update()   // Logic: physics, collisions
  game.draw()     // Render: redraw everything
  requestAnimationFrame(loop)  // Schedule next
}
loop()  // Start!
requestAnimationFrame
Browser API that calls your function right before the next screen repaint. Automatically pauses when the tab is hidden, saving CPU/GPU.

At 60 fps each frame is ~16.67 ms. All update + draw logic must finish within this budget.

📄 main.js — Entry Point

import Game from "./Game.js"

const canvas = document.getElementById("game")
const ctx = canvas.getContext("2d")

// Create Web Worker in a separate thread for AI/prediction
const worker = new Worker("./js/GameWorker.js")

const game = new Game(canvas, ctx, worker)

function loop() {
  game.update()
  game.draw()
  requestAnimationFrame(loop)
}

loop()
Key point: The Worker is created here and passed into Game. This separates concerns: main thread handles rendering, worker handles AI.

🎮 Game.js — The Orchestrator

Responsibilities

  • Manages Player, Obstacles[], LevelManager
  • Spawns obstacles at intervals based on current level
  • Runs collision detection every frame
  • Draws all entities + HUD (score, level)
  • Sends canvas snapshots to the Worker every 200 ms
  • Handles game over & restart

Constructor

constructor(canvas, ctx, worker) {
  this.canvas = canvas
  this.ctx = ctx
  this.worker = worker
  this.bg = new Image()
  this.bg.src = "./assets/bg.png"
  this.player = new Player()
  this.obstacles = []
  this.frame = 0
  this.score = 0
  this.running = true
  this.levelManager = new LevelManager()
  new Input(this.player)
  this.setupWorker()
}

🎮 Game.js — update()

update() {
  if (!this.running) return
  this.frame++

  this.player.update()  // Apply gravity

  // Spawn new obstacle every spawnRate frames
  if (this.frame % this.levelManager.spawnRate() === 0) {
    this.spawn()
  }

  // Iterate backwards (safe removal with splice)
  for (let i = this.obstacles.length - 1; i >= 0; i--) {
    let o = this.obstacles[i]
    o.update(this.levelManager.speed())

    if (collide(this.player, o)) {
      this.gameOver()
    }

    if (o.x + o.w < 0) {       // Off-screen → remove + score
      this.obstacles.splice(i, 1)
      this.score++
      this.levelManager.update(this.score)
    }
  }
}
Why iterate backwards? When using splice(i, 1) to remove an element, iterating backwards (i--) prevents skipping the next element.

🎮 Game.js — draw()

Redraws the entire scene every frame. Order matters — later draws appear on top.

draw() {
  this.ctx.clearRect(0, 0, this.canvas.width, this.canvas.height)  // 1. Clear

  if (this.bg.complete)  // 2. Background
    this.ctx.drawImage(this.bg, 0, 0, this.canvas.width, this.canvas.height)

  this.player.draw(this.ctx)                    // 3. Player
  for (let o of this.obstacles) o.draw(this.ctx) // 4. Obstacles

  this.ctx.fillStyle = "white"                  // 5. HUD
  this.ctx.font = "20px Arial"
  this.ctx.fillText("Score: " + this.score, 10, 25)
  this.ctx.fillText("Level: " + this.levelManager.level, 10, 50)

  if (!this.running) this.drawGameOver()        // 6. Game Over overlay
}
Canvas 2D rendering is an immediate-mode API — nothing is retained between frames. You must redraw everything from scratch each time.

🎮 Game.js — Worker Communication

setupWorker() {
  this.worker.onmessage = (e) => {
    if (e.data.type === 'ready')  console.log('[Game] Worker ready')
    if (e.data.type === 'action' && e.data.action === 'jump')
      this.player.jump()
  }

  // Send canvas snapshot to Worker every 200ms
  this.snapshotInterval = setInterval(async () => {
    if (!this.running) return

    const bitmap = await createImageBitmap(this.canvas)
    this.worker.postMessage(
      { type: 'predict', image: bitmap }, [bitmap]  // Transferable!
    )

    this.worker.postMessage({
      type: 'gameState',
      state: { player, obstacles, score, level }
    })
  }, 200)
}
Transferable Objects: [bitmap] in the second argument transfers ownership of the ImageBitmap to the Worker — zero-copy, no serialization overhead.

🐤 Player.js — The Bird

class Player {
  constructor() {
    this.x = 80       // Fixed horizontal pos
    this.y = 200      // Starting Y
    this.w = 60; this.h = 40  // Size
    this.vy = 0       // Vertical velocity
    this.sprite = new Image()
    this.sprite.src = "./assets/player.png"
  }

  jump() { this.vy = -10 }

  update() {
    this.vy += 0.5     // Gravity
    this.y += this.vy   // Apply velocity
    if (this.y < 0) this.y = 0
    if (this.y > 350) { this.y = 350; this.vy = 0 }
  }

  draw(ctx) {
    if (this.sprite.complete)
      ctx.drawImage(this.sprite,
        this.x, this.y, this.w, this.h)
  }
}

Physics Model

Euler Integration (simplest physics simulation):

velocity += gravity → acceleration
position += velocity → movement

Each frame, vy increases by 0.5 (gravity pulls down). jump() sets vy = -10 (negative = upward).
Canvas coordinate system: Y=0 is the top. Y increases downward.
So vy = -10 means "go up".

🪨 Obstacle.js

class Obstacle {
  constructor(canvas) {
    this.w = 50
    this.h = Math.random() * 120 + 60
    //     → random height: 60–180px

    this.x = canvas.width
    //     → spawns at right edge

    this.y = Math.random()
            * (canvas.height - this.h)
            - 30
    //     → random vertical position

    this.sprite = new Image()
    this.sprite.src =
      "./assets/obstacle.png"
  }

  update(speed) {
    this.x -= speed  // Move left
  }

  draw(ctx) { /* drawImage(...) */ }
}

Obstacle Lifecycle

  1. Spawn — Created at x = canvas.width (right edge)
  2. Move — Each frame: x -= speed
  3. Collide? — Checked by collide(player, obstacle)
  4. Off-screen — When x + w < 0 → removed, score++

Design choice: The player stays at a fixed X position. Obstacles move toward the player, creating the illusion of forward movement. This is a classic side-scroller pattern.

💥 Collision.js — AABB Detection

export default function collide(a, b) {
  return (
    a.x < b.x + b.w &&  // A left < B right
    a.x + a.w > b.x &&  // A right > B left
    a.y < b.y + b.h &&  // A top < B bottom
    a.y + a.h > b.y     // A bottom > B top
  )
}
If all four conditions are true → overlap → collision!

AABB — Axis-Aligned Bounding Box

  • Axis-Aligned = edges parallel to X/Y axes
  • Bounding Box = smallest enclosing rectangle
  • O(1) — just 4 comparisons
  • Used in Unity, Unreal, Godot, and virtually every 2D engine

For rotated objects → OBB or SAT (Separating Axis Theorem).

📈 LevelManager.js — Difficulty Progression

class LevelManager {
  constructor() {
    this.level = 1
    this.levels = [
      { speed: 3,   spawn: 120 }, // L1
      { speed: 3.5, spawn: 110 }, // L2
      { speed: 4,   spawn: 100 }, // L3
      { speed: 4.5, spawn:  90 }, // L4
      { speed: 5,   spawn:  80 }, // L5
      { speed: 5.5, spawn:  70 }, // L6
      { speed: 6,   spawn:  60 }, // L7
      { speed: 7,   spawn:  50 }, // L8
    ]
  }
  speed()     { return this.levels[this.level-1].speed }
  spawnRate() { return this.levels[this.level-1].spawn }
  update(score) {
    if (score % 10 === 0 && this.level < 8)
      this.level++
  }
}
LevelSpeedSpawn≈ Interval
13 px/f120 f~2.0 s
44.5 px/f90 f~1.5 s
87 px/f50 f~0.8 s
Every 10 points → level up.
Higher level = faster obstacles + more frequent spawns. This is a common difficulty curve pattern.

🔍 What is YOLO? — History

  • 2015 — YOLOv1 by Joseph Redmon. Revolutionary paper: "You Only Look Once"
  • 2016-2020 — v2, v3 by Redmon; v4 by Bochkovskiy et al.
  • 2021 — YOLOv5 by Ultralytics (Glenn Jocher). First YOLO in PyTorch (not Darknet C).
  • 2023-2025 — YOLOv8, v9, v10, v11 by various teams
"You Only Look Once" — Unlike earlier methods (R-CNN) that scanned images multiple times, YOLO processes the entire image in a single forward pass.

How It Works (Simplified)

  1. Divide the image into an S×S grid
  2. Each cell predicts B bounding boxes + confidence
  3. Each cell also predicts C class probabilities
  4. Non-Maximum Suppression (NMS) removes overlapping boxes

🔧 TensorFlow.js (TFJS)

What is it?

  • Google's ML library for JavaScript
  • Run / train / convert ML models in the browser or Node.js
  • Backends: WebGL (GPU), WASM, CPU
  • Loads models from PyTorch, TF, Keras, ONNX

Key Concepts

Tensor — Multi-dimensional array.
[3] → 1D   [3, 4] → 2D   [1, 640, 640, 3] → 4D
Memory management: Tensors live on the GPU. You must manually dispose() them or wrap code in tf.tidy() to avoid memory leaks!

⚙️ Web Workers — Why & How

The Problem

JavaScript is single-threaded. Running a neural network (50-200ms) on the main thread would freeze the game loop.

The Solution

Web Workers run JS in a separate OS thread, communicating via postMessage().

Workers cannot: access DOM, window, or canvas. Messages only.

Communication Flow

Main Thread          Web Worker
    │──── predict ────▶  │
    │    (ImageBitmap)    │ ← runs YOLOv5
    │◀── prediction ────  │
    │    (bounding boxes) │
    │──── gameState ──▶  │
    │    (positions)      │ ← decides action
    │◀── action ────────  │
    │    (jump/none)      │

🧠 GameWorker.js — Model Loading

importScripts('https://cdn.jsdelivr.net/npm/@tensorflow/tfjs@latest')

const MODEL_PATH = '../machine-learning/yolov5n_web_model/model.json'
const LABELS_PATH = '../machine-learning/yolov5n_web_model/labels.json'
const INPUT_MODEL_DIMENSIONS = 640
const CLASS_THRESHOLD = 0.5

async function loadModelYOLOv5AndItsLabels() {
  await tf.ready()                                    // 1. Init TF backend

  _labels = await (await fetch(LABELS_PATH)).json()   // 2. Load class names
  _model = await tf.loadGraphModel(MODEL_PATH)        // 3. Load model

  // 4. Warmup — compile WebGL shaders, pre-allocate GPU memory
  const dummyInput = tf.ones(_model.inputs[0].shape)
  await _model.executeAsync(dummyInput)
  tf.dispose(dummyInput)

  postMessage({ type: 'model-loaded' })               // 5. Signal ready
}
Warmup explained: The first inference is always slow because WebGL needs to compile shaders and allocate GPU buffers. Running a dummy inference at startup ensures the real predictions are fast.

🧠 GameWorker.js — Image Preprocessing

function createAndFormatImageTensor(image) {
  return tf.tidy(() => {                                // Auto-dispose intermediates
    const tensorImage = tf.browser.fromPixels(image)    // [400, 800, 3] H×W×RGB
    return tf.image
      .resizeBilinear(tensorImage, [640, 640])          // [640, 640, 3]
      .div(255)                                         // Normalize to [0, 1]
      .expandDims(0)                                    // [1, 640, 640, 3]
  })
}
ImageBitmap
800×400 → fromPixels
[400,800,3] → resize
[640,640,3] → ÷ 255
0–1 → expandDims
[1,640,640,3]
tf.tidy() auto-disposes all intermediate tensors in the callback, except the returned one. Essential for preventing GPU memory leaks.

🧠 GameWorker.js — Running Inference

async function runYoloModelOverTensorAndGetDetections(tensor) {
  const output = await _model.executeAsync(tensor)     // 1. Run model
  tf.dispose(tensor)                                   // 2. Free input

  const [boxes, scores, classes] = output.slice(0, 3)  // 3. Extract outputs

  const [boxesData, scoresData, classesData] =         // 4. GPU → CPU
    await Promise.all([boxes.data(), scores.data(), classes.data()])

  output.forEach(t => t.dispose())                     // 5. Free outputs
  return { boxes: boxesData, scores: scoresData, classes: classesData }
}
Model outputs (YOLOv5): boxes [x1,y1,x2,y2] · scores confidence 0–1 · classes integer index → labels.json

🧠 GameWorker.js — Processing Predictions

// Generator function — yields predictions one at a time
function* processPrediction({ boxes, scores, classes }, width, height) {
  for (let index = 0; index < scores.length; index++) {
    if (scores[index] < CLASS_THRESHOLD) continue  // Filter low confidence

    const label = _labels[classes[index]]
    console.log(`[Worker] Detection: ${label} (confidence: ${scores[index].toFixed(2)})`)
  }
}
Generator Functions (function*)
A special JavaScript function that can pause and resume execution. Uses yield to emit values lazily, one at a time. No intermediate array is created, saving memory.
CLASS_THRESHOLD = 0.5
Only detections with confidence ≥ 50% are kept. This filters out noise and false positives. In production, you'd tune this based on precision/recall requirements.

🧠 GameWorker.js — Message Handler

self.postMessage({ type: 'ready' })   // Signal: worker is alive

self.onmessage = async (e) => {
  const { type } = e.data

  if (type === 'predict') {
    if (!_model) return   // Model not loaded yet

    const input = createAndFormatImageTensor(e.data.image)
    const { width, height } = e.data.image

    const results = await runYoloModelOverTensorAndGetDetections(input)

    for (const prediction of processPrediction(results, width, height)) {
      postMessage({ type: 'prediction', ...prediction })
    }
  }

  if (type === 'gameState') {
    const state = e.data.state
    // TODO: Use game state (player pos, obstacles, score)
    // to decide whether to jump
  }
}

loadModelYOLOv5AndItsLabels()
Two input channels: predict = visual (image frame)  |  gameState = numerical (positions, score). Combining both would enable a more intelligent agent.

🎨 Key Design Decisions

✅ Good choices
  • Web Worker for AI — main thread stays at 60 fps
  • Transferable ImageBitmap — zero-copy frame transfer
  • YOLOv5n — smallest model, best for real-time browser use
  • tf.tidy() + dispose() — disciplined GPU memory management
  • Generator for predictions — lazy, memory-efficient iteration
  • ES Modules — clean separation, no build step needed
🔮 Future opportunities
  • Use gameState data + detections to build an autonomous agent
  • Fine-tune YOLOv5 on actual game sprites for better detection
  • Add OffscreenCanvas for drawing in the Worker
  • Use Reinforcement Learning (Q-learning / NEAT) for decision-making
  • Implement actual bounding box overlay for debug visualization

🤖 How Could The AI Actually Play?

The Worker detects objects but doesn't decide actions yet. Two approaches:

Approach 1: Rule-Based

if (type === 'gameState') {
  const { player, obstacles } = state
  const nearest = obstacles
    .filter(o => o.x > player.x)
    .sort((a, b) => a.x - b.x)[0]
  if (nearest) {
    const gap = nearest.y + nearest.h / 2
    if (player.y > gap + 20)
      postMessage({ type:'action', action:'jump' })
  }
}
